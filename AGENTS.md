## What Hardwise is

A server-hardware pre-layout review trust layer: ingest Cadence/Allegro exported netlist/PST + BOM + optional Capture pin table + public datasheet/profile evidence, run deterministic validators and a tool-use loop, then output a review workbench/report where every claim carries a source token and every reference designator is registry-verified.

KiCad remains an appendix path for public reproducibility, regression tests, and evidence-chain demos. It is not the product-default priority when deciding new roadmap work.

Architecture inspired by [Wrench Board](https://github.com/Junkz3/wrench-board) (Anthropic *Build with Opus 4.7* hackathon, 2nd place, April 2026) — **design ideas only, no code copied**.

## Product direction

Hardwise should win as the review trust layer around existing enterprise EDA exports, not as an auto-layout or auto-routing system. Before setting any long-term roadmap goal, compare against current EDA/AI hardware tools and choose the gap that strengthens server hardware review trust: Cadence/Allegro intake, BOM identity, Capture pin evidence, public datasheet/profile evidence, deterministic validation, and auditable Copilot traces.

Industry research belongs in `docs/server_hardware_roadmap.md` or `docs/rolling_log.md`; permanent product constraints belong here. Do not promote a KiCad parser, PCB layout automation, SI/PI simulation, DFM automation, PLM/BOM lifecycle, or live Cadence plugin idea to the default goal unless fresh external research shows it is the highest-leverage trust-layer gap.

## Hard rules — NEVER violate

1. **No company-internal hardware data.** Author is a working hardware engineer at an SOE; this project uses only public/synthetic EDA exports, public KiCad fixtures, public datasheets, and hand-authored public profiles. Push back on any proposal to use internal data, even sanitized — demo must be reproducible from public sources only.
2. **Wrench Board is reference, not source.** License is `NOASSERTION` (source-available proprietary). Architecture ideas can be borrowed and attributed publicly; code cannot be copied or transcribed. Adopting Wrench Board's MIT/Apache/BSD dependencies is fine — they are independent of the Wrench Board license.
3. **No hallucinated refdes — defense in depth, two layers.** Every refdes the agent surfaces must originate from a tool lookup against the EDA registry, never free-generated by the model.
   - **Layer 1 — tool discipline:** the runtime agent surface is 5 core query tools (`list_components`, `get_component`, `get_nc_pins`, `search_datasheet`, `run_component_validation`) plus 7 context/topology tools in the workbench path (about 12 tools total). Tools return structured misses for unknown / unconfigured inputs. Refdes-specific lookups such as `get_component`, `run_component_validation`, and context lookups include `closest_matches`; tools never fabricate.
   - **Layer 2 — sanitizer:** every outbound message text is regex-scanned for refdes-shaped tokens (`\b[A-Z]{1,3}\d{1,4}\b`), validated against the parsed-board registry, and unverified matches are wrapped as `⟨?U999⟩` before reaching the user. A refdes-shaped token that is itself a parsed component identity (a part number or package such as `EG2132` / `SOP8`) is verified-from-source and passes through; only tokens absent from the board are wrapped. Implementation lives at `src/hardwise/guards/refdes.py`.
   Architecture borrowed from Wrench Board's `api/agent/sanitize.py`.
4. **Two-week MVP scope.** Demo-able loop, not feature-complete product. If a task is stuck past 4 hours → propose downscope, don't grind.
5. **Demo anchor: schematic-review node only.** Hardwise is bound to one workflow node — pre-Layout schematic review. Any cross-node feature (PCB review, simulation, test-result feedback loops, BOM management, FMEA, PLM) is out of scope. See `docs/review_node.md` for the full node profile and `docs/PLAN.md` DR-007 for the rationale.
6. **Completion requires evidence.** Any agent claiming "done" must attach test output (`uv run pytest -q`) and commit hash. A claim without evidence is treated as incomplete. This applies especially to overnight tasks and cross-session handoffs.
7. **Options before plans.** For non-trivial tasks spanning multiple files or introducing architectural choices, present 2-3 solution approaches with trade-offs before starting implementation. Let the human choose the direction, then proceed with a plan.

## Stack

- **Runtime:** Python 3.11+, uv (package + venv), python-dotenv
- **Agent:** `anthropic >= 0.40.0` — Anthropic-format API; `ANTHROPIC_BASE_URL` overrides the endpoint (third-party proxy supported natively)
- **CLI:** `typer >= 0.12.0`
- **Schema:** `pydantic >= 2.0.0` — every tool input/output is a Pydantic model (runtime validation + JSON Schema source)
- **Test/lint:** `pytest >= 8.0.0`, `ruff >= 0.6.0`
- **Heavy deps (installed per slice, not upfront):** `chromadb` (vector store, local mode), `sentence-transformers` (local embeddings), `pdfplumber` (datasheet text extraction) — added at Slice 3 when datasheet ingest first lands; `sqlalchemy >= 2.0` + SQLite (relational store) added the same slice. KiCad parsing is a hand-rolled appendix/regression path in `adapters/kicad.py`, no `kiutils` dependency.

## Models

The agent reads three tier slots at runtime (`HARDWISE_MODEL_FAST/NORMAL/DEEP` from `.env`). The slots are **decoupled from how many distinct upstream models exist** — if the upstream API exposes only one model, all three may resolve to it; if it exposes a fast/normal/deep variant family, each slot maps to one. The agent code never names a specific model.

Default mapping (per `.env.example`):

| Env var | Upstream model | Tier | Used for |
|---|---|---|---|
| `HARDWISE_MODEL_FAST` | `mimo-v2.5` | fast | intent classification, simple lookups, single-tool extractions |
| `HARDWISE_MODEL_NORMAL` | `mimo-v2.5` | normal | review main loop, tool orchestration, evidence assembly |
| `HARDWISE_MODEL_DEEP` | `mimo-v2.5` | deep | hard reasoning only — multi-step diagnosis, ambiguous datasheet interpretation |

**Upstream**: MiMo (Xiaomi), served via the `xiaomimimo.com` proxy that speaks Anthropic message format on `/anthropic/*`. The `anthropic` Python SDK is a generic Anthropic-protocol client; the upstream identity is set by `ANTHROPIC_BASE_URL` and the model name in the `messages.create` call. **API model identifiers are lowercase** (`mimo-v2.5`, not `MiMo-V2.5`); list available models with `curl -H "x-api-key: $KEY" https://token-plan-sgp.xiaomimimo.com/v1/models`. A `mimo-v2.5-pro` variant exists if a real "deep" tier is later needed.

Tier selection lives in `src/hardwise/agent/router.py`. Default for any review run is `normal`; `deep` must be explicitly opted into per task. `src/hardwise/agent/runner.py` consumes the router when running the tool-use loop.

Current agent tool surface: 5 core query tools (`list_components`, `get_component`, `get_nc_pins`, `search_datasheet`, `run_component_validation`) plus 7 context/topology tools for workbench runtime (component/net context, net search, project topology summary, document coverage, and scoped evidence lookup), for about 12 tools total. The full manifest lives in `src/hardwise/agent/tools.py`.

## Workbench (Allegro Copilot)

Two entrypoints render the Allegro project workbench with an optional Copilot panel; both reuse the same deterministic `validation/` truth and the same Refdes Guard.

- `design-validator-ui <netlist/pst> <bom> --ai-snapshot` — single static HTML with baked, audited chat transcripts; opens via `file://`, no server, no API key. Without `--ai-snapshot` the static workbench is unchanged.
- `serve-workbench <netlist/pst> <bom>` — local FastAPI server exposing `POST /api/workbench/chat`; `--fake-ai` runs a deterministic fake Anthropic client, real mode reads `.env`. The browser never receives API keys.

Trust invariants: detailed contracts and test requirements live in `.trellis/spec/backend/workbench-guidelines.md`. Key invariants (do not regress):
- `--fake-ai` drives the **real** `Runner` + real tool dispatch + guard; the fake client only emits `tool_use`/text. It never bypasses the Runner.
- Allegro reaches the agent via a `Design → BoardRegistry` shim populated into an in-memory relational `Session` (`workbench/context.py`); the Runner `registry`/`session` contract and the guard are unchanged.
- Live mode keeps the same 5-core + 7-context runtime tool surface. Without a vector collection, `search_datasheet` returns its structured not-configured result.

Modules: `workbench/{context,chat,server}.py`, `report/copilot_panel.py` + `report/copilot_panel_assets.py`. Server deps: `fastapi` + `uvicorn` (`serve-workbench` only).

```bash
uv sync                       # install deps
uv run hardwise --help        # CLI
uv run pytest -q              # tests (fast subset)
uv run ruff check .           # lint
uv run ruff format .          # format
```

Single test / subset:

```bash
uv run pytest tests/guards/test_refdes.py -v
uv run pytest -k "not slow"
```

Tests do not require `ANTHROPIC_API_KEY`; only runtime code paths raise if it's missing. Tests that hit the live API or take longer than ~1 s should be marked `@pytest.mark.slow` so they only run in `pytest -m slow` or `pytest` (full).

## Conventions

- **Explain before code** in hardware-engineer language. Any new term → ≤100-char gloss inline.
- **Minimal closures.** No premature abstractions, no speculative features, no half-finished implementations.
- **One module / one function at a time.** Don't dump large blocks. Each shipped module → one-paragraph blurb usable in README / 简历 / 面试.
- **Tools return structured null/unknown, never fabricated.** Every Pydantic tool output has a `found: bool` discriminator (or equivalent). Unknowns return `{"found": false, "reason": "...", "closest_matches": [...]}`. The agent picks recovery; tools never invent.
- **Datasheet profile vs board policy.** Ready `DatasheetProfile` records only device-intrinsic public datasheet facts: pin definitions, absolute-maximum ratings, recommended operating ranges. Board-level engineering policies such as bootstrap reverse-voltage margin, decoupling values, trace-width rules, or topology-derived requirements live in validators and are derived from schematic/netlist context, not pre-stored as profile JSON facts.
- **File size guard.** If a file creeps past ~300 lines, ask whether it should split. One responsibility per module, no god classes.
- **Verify before declaring done.** Run `uv run pytest -q` and `uv run ruff check .` before saying a change is complete. For CLI changes, run the command at least once.
- **Learning log discipline.** Every bug, framework quirk, or surprise gets a `docs/learning_log.md` entry: Symptom / Root cause / Fix / Takeaway. Debugging is learning.
- **Iterate the FAQ.** When a module ships, update `docs/faq.md` with the new measured fact; keep it public-facing and free of interview framing.
- **Two stores, one join key.** Datasheet text → vector store; refdes / nets / BOM / DRC → relational. Never mix shapes between stores; refdes is the join key when crossing.

## Commit hygiene

- **Conventional commits**: `feat(scope):` / `fix(scope):` / `refactor(scope):` / `chore(scope):` / `docs(scope):` / `test(scope):`. Scope is usually a module name (`refdes`, `agent`, `kicad`).
- Each commit must pass `uv run pytest -q` and `uv run ruff check .` independently.
- **Never `git push` without explicit authorization.** Local commits are encouraged; pushing to `origin` requires confirmation.
- **Never rewrite published history** (`reset --hard`, `rebase -i`, `commit --amend` after share). Land a follow-up commit instead.
- Don't bundle changes from two different domains (e.g. `guards/` + `agent/`) into the same commit even if landed in the same session — stage narrowly across module boundaries, commit cohesively within a module.

## API configuration

The upstream is an **Anthropic-format API** — the protocol, not necessarily the vendor. The `anthropic` Python SDK reads both `ANTHROPIC_API_KEY` and `ANTHROPIC_BASE_URL` natively, so no custom client construction is needed. Real values live in `.env` (gitignored); template lives in `.env.example`.

Default upstream is the `xiaomimimo.com` Anthropic-compatible proxy serving the MiMo model family (see Models above). Pointing `ANTHROPIC_BASE_URL` at a different endpoint (Anthropic's own API, a self-hosted gateway, etc.) requires no code change.

Verify connectivity end-to-end: `uv run hardwise verify-api` — sends a one-shot prompt and prints the response + token counts.
