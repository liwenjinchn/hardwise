01

Submission Boundary

A public KiCad schematic can be parsed into a trusted registry; deterministic checks can produce evidence-gated findings; and the agent can answer schematic questions only through structured tools.
Review reportR001,R002,R003 产出 registry-verified refdes 和 evidence tokens。

Quality gatepytest 与 ruff 通过。

R004/R005、schematic-side net parser、GitHub Action、gold-label eval、Cadence/Allegro adapter、PR-comment workflow 全部是 post-MVP。
02

Sprint Goal

  1. 读取公开 KiCad project 和 active schematic-review rules。
  2. 运行 deterministic checks 和 Anthropic-format tool-use loop，board objects 只来自 structured tools。
  3. 产出 markdown/HTML review report，每个 refdes registry-verified，每个 finding 有 source token。
  4. 把 candidate review rules 写入 memory/rules.md，并保留 human gate。

05

Decision Records

DR-001 Direct mode

用 messages.create + Python tool loop，不上 Managed Agents。MVP review sessions 是 stateless。

DR-002 KiCad-only adapter

Mac 本地无法跑 Cadence，公开样例以 KiCad 为主。Cadence/Allegro 是后续新 adapter。

DR-003 SQLite + Chroma

零运维、本地可复现。SQLite/Chroma 证明关系库 + 向量库的架构模式。

DR-004 Plan mode workflow

多文件或陌生实现先 plan，再执行，避免 scaffold 早期 Typer bug 那类粗心问题。

DR-005 MiMo via Anthropic format

模型槽位来自环境变量，代码不写死具体模型名。

DR-006 Vertical slice

每个 slice 都要有端到端可运行物，避免水平铺底到第 8 天还没东西可演示。

DR-007 Review node anchor

只锚定 pre-layout 原理图评审节点，PCB/仿真/PLM 等跨节点能力延后。

DR-008 Finding schema

Slice 1 就立统一 Finding，R001 必须读 schematic-only raw records。

DR-009 Evidence chain + decision

decision 是规则判断，status 是人工流程状态；两者严格分离。
