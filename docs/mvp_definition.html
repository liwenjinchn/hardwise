Hardwise MVP 定义

Hardwise 的 MVP 是一个可信的 Layout 前原理图评审工作台。它不证明大模型能独立判断完整硬件设计；它证明一个更窄、更有用的流程：公开 EDA 文件、BOM 身份和公开规格书/器件档案证据进入评审工作台，每个对象可追溯，每个不确定项留给评审人。

评审节点交给 Layout 前的原理图评审。
可信边界位号来自器件台账；评审发现必须带证据出处。
AI 的角色AI 助手解释工具证据，不替代确定性结论。

用户问题

目标用户是准备把原理图交给 Layout 的硬件工程师。评审前他们需要快速知道：哪些器件必须看、哪些只是人工缺口、每条结论由哪一页规格书、哪条网表事实、哪条规则或哪行 BOM 支撑，以及 AI 有没有提到真实存在的板上对象。

核心流程

  1. 导入公开原理图工程，或导出的原理图网表 + BOM。
  2. 建立位号、引脚、网络、BOM 身份的可信器件台账。
  3. 匹配本地公开规格书索引和人工审核过的结构化器件档案。
  4. 运行确定性检查清单规则和器件族验证器。
  5. 把输出分成 必须修人工确认检查已满足。
  6. AI 助手只解释有工具支撑的事实：器件上下文、验证结果、连接拓扑、规格书证据、未知位号未命中。
  7. 导出评审工作台和反馈清单，供原理图评审会逐项关闭。

页面结构

  • 项目摘要：输入文件、器件数、BOM 配对数、已验证数、通过/警告/错误/人工计数。
  • 评审队列：问题优先的"必须修 / 人工确认 / 检查已满足"行。
  • 器件详情：身份、引脚、网络、验证结果、档案/文档状态、证据出处。
  • AI 助手面板：展示工具调用轨迹和可信度档位，解释结果但不覆盖 L1 结论。
  • 反馈导出：位号、描述、证据、评审人接受/拒绝、关闭状态。

范围内

公开 KiCad 工程、公开 Allegro 网表/PST + 原理图 BOM 样例、台账校验过的器件/引脚/网络/BOM 身份、公开规格书与器件档案证据、确定性规则、人工审核过的器件验证器、L1/L2/L3 可信度分层、静态 HTML 工作台和本地 AI 助手面板。

非目标

不做 PCB layout、.brd、boardview、布局布线、SI/PI、EMC、热、仿真、PLM、供应商风险、价格/库存、生产 BOM 治理、公司内部硬件数据、整板自动正确性判断、器件档案草稿自动转就绪。

验收标准

  • 公开演示输入能打开含项目摘要、评审队列、器件详情、证据和人工缺口的工作台。
  • 未知位号如 U999 不能被展示成有效板上对象。
  • 每条硬结论都有证据出处；没有证据就降级。
  • 至少一个 L1 确定性问题能从器件追到网络/档案证据。
  • 至少一个 L2 规格书回答展示 datasheet:<file>#pN 证据出处。
  • AI 助手回答暴露工具轨迹，不能覆盖 L1 验证器结果。
