# Hardwise 常见技术问题

> 这份 FAQ 回答关于 Hardwise 设计取舍的六个高频问题。所有数字都可以用
> README 里的命令在公开数据上复现。

## Q1. 这个工具帮硬件工程师省了哪一步？

省的是评审前后的"搬运链"：在原理图、BOM、规格书、引脚定义和检查清单之间
反复查证、对位号、整理证据、写结论。

Hardwise 把这条链整理成评审工作台：导入网表 + BOM 后自动建立器件台账，跑
确定性检查，把结果分成"必须修 / 人工确认 / 检查已满足"三类队列，每条结论
带证据出处。工程师省掉的是查位号、对引脚、整理证据、排除模型胡编对象的
时间；**最终是否接受每条意见，仍由人来判断**。

确定性检查的选型原则：优先覆盖人工评审里最常因人力成本被整体放弃的
体力活——例如逐 pin 对照规格书核对 NC 处理。人工因成本放弃的检查，
恰是自动化回报最高的。

它不替代硬件工程师下判断，也不宣称大模型可以独立评审完整硬件设计。

## Q2. 输入是什么，输出是什么？

**输入（全部公开数据）：**

- Cadence/Allegro 导出的原理图网表（Telesis 单文件或 PST 三件套）+ 导出 BOM
  （CSV/TSV/Cadence `.BOM`/窄列 `.xlsx`）——工作台主路径
- 可选 Capture pin-table CSV——补充网表/BOM 没有的 pin electrical type、
  NC marker 和页面位置；当前只作为 R008/R009/R010 的 L1 review task 来源
- KiCad 工程目录（`.kicad_sch` + `.kicad_pcb`）——规则评审与回归基线路径
- 公开规格书 PDF/HTML + 人工审核过的结构化器件档案（`DatasheetProfile`）
- 检查清单规则（`data/checklists/sch_review.yaml`）

**输出：**

- 三栏评审工作台（本地服务或离线单文件 HTML），含器件队列、验证明细、
  证据出处和 AI 助手问答轨迹
- Markdown/HTML 评审报告与可导出的评审反馈清单
- SQLite/PostgreSQL 结构化存储与运行轨迹台账

公开演示数字：mixed controller 样例 25 个器件 / BOM 全配 / 22 行确定性验证 /
通过-警告-错误 = 5-13-4 / 3 行人工缺口；KiCad `pic_programmer` 121 个器件 /
77 个 NC 引脚 / 19 个显式 schematic-side 命名网络 / 29 条带证据评审发现。
pin-table 检查进入工作台队列，但不改写这组 PASS/WARN/ERROR 总账。

## Q3. 哪些数据进向量库，哪些进关系库？为什么这样分？

- **关系库（SQLAlchemy，SQLite 与 PostgreSQL 同套表结构）**：位号、引脚、
  网络、BOM 身份、审核过的器件档案——强 schema、要做 join 和位号校验的数据。
- **向量库（Chroma 本地持久化）**：规格书长文本切块——无固定 schema、需要
  语义检索的数据。每个切块带 `{来源 PDF, 页码, 器件}` 出处元数据。

**位号 + 型号是两库之间的关联键。** 混存会让位号查询和参数引用都失去可信度：
规格书文本进关系库没法做语义检索，位号进向量库会失去精确校验。

实测链路：`l78.pdf` 切成 157 块入 Chroma，检索"absolute maximum input
voltage"top-1 命中第 4 页；评审规则引用的是人工审核过的档案事实
`datasheet:l78.pdf#p4`，向量检索是对同一页的独立佐证。

## Q4. AI 助手有哪些工具？为什么不让模型自由回答？

五个工具，输入输出全部经 Pydantic schema 校验：`list_components`、
`get_component`、`get_nc_pins`、`search_datasheet`、`run_component_validation`。

模型自由回答会编造位号、网络名和参数。强制走工具意味着每条回答都有查询
和返回值，可审计、可复现。关键设计在"未命中"分支：问一个不存在的 `U999`，
工具不会编一个，而是返回 `found=false` 加上最接近的真实位号候选；没有加载
设计或没有器件档案时，验证工具返回"未配置 / 无档案"，而不是编一个结论。

回答按三档可信度标注：

- **L1 确定性**——通过/警告/错误结论只能来自 Python 规则和验证器，模型可以
  解释它，不能推翻它。
- **L2 有出处**——规格书问答只有携带页码级检索证据（如 `datasheet:l78.pdf#p4`）
  才标这一档。
- **L3 人工确认**——没有档案、没有检索证据，就留在人工确认区。

## Q5. 怎么防止编造位号和规格书参数？

三层合同，不是一句提示词：

1. **工具层**——未知位号返回结构化未命中 + 候选项，工具从不编造。
2. **输出层（位号防护）**——每条对外文本都用正则扫描位号形状的词
   （`U1`/`R10`/`J5`），对照解析出的器件台账校验，未注册的包裹成 `⟨?U999⟩`
   再显示。
3. **报告层（证据台账）**——没有 `sch:` / `datasheet:` / `rule:` 出处标记的
   评审发现，写报告前直接丢弃，报告头公示丢弃计数。

回归基线持续盯防：5 个公开仓库 / 6 个 KiCad 工程 / 1707 个器件 / 437 条
评审发现的公开评测包里，"未验证位号包裹数"和"因缺证据被丢弃的发现数"两个
硬指标都锁定为 0 才算通过。

## Q6. 它明确不做什么？

- 不做 PCB layout、SI/PI、EMC、热、仿真和时序收敛
- 不解析 `.brd`、boardview、布局布线和 PCB 几何
- 不做 PLM、生命周期、价格、库存和供应商风险
- 不做实时 Cadence/Allegro 插件——消费的是导出文件
- 不用任何公司内部硬件数据，即使脱敏
- 评测包是回归冒烟，不包装成专家准确率基准

已知限制与后续方向：人工标注的判定准确率校准集、原理图侧网络拓扑解析器
（解锁 fanout / 连接端点类规则；网名级的命名策略检查已能读取 KiCad `.kicad_sch`
显式标签，也已在 Allegro 网表路径落地）、
企业 EDA 环境适配。原则是先窄而真，不为覆盖率把弱证据包装成硬结论。
