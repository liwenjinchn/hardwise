不是泛泛聊天，而是把输出压进五个机制和三层信任。

01

五个机制锁边界

Refdes Guard、Evidence Ledger、Sleep Consolidator、Tiered Routing 和 Prompt Caching 分别约束位号、证据、规则演化、模型选择和长上下文成本。

02

三层 trust 标证据

L1 deterministic 才能给 PASS/WARN/ERROR；L2 grounded 只表示本轮有页码级 retrieval；L3 manual 明确留给 reviewer。

03

两条公开轨支撑

KiCad 轨证明 agent/review/evidence guard；Allegro 轨证明项目级 workbench 和多 family validator，全程不用公司内部硬件数据。

面试时可以这样讲

先讲五大机制和 L1/L2/L3，再展示输入、输出和边界：哪些判断已经 deterministic，哪些问题仍然是 manual/no-profile。

面试官它怎么发现 U12 的问题？
Hardwise先匹配 XL1509 profile，再从 netlist 看到 OUTPUT 接 D5 和 L1，确定 D5=1N4007W 不是 Schottky-style，L1=6.8uH 低于推荐范围。
面试官它是不是自动评审全 BOM？
Hardwise不是。当前只验证有本地 profile 的 4 个器件；其余 21 个在 project index 中显式显示为 no_profile/manual。
