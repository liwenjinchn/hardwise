00

结论

Hardwise 当前不是“没做完”，而是已经有可演示闭环，但叙事被支线工程稀释。

Hardwise 不是证明“大模型能自动评审硬件”，而是证明“硬件评审 Agent 必须被 EDA registry 和 evidence token 约束；模型只能解释工具查到的对象，不能自由发明工程事实。”

接下来不再用功能数量证明项目价值。用边界感、证据链、反幻觉约束证明项目价值。

02

主线资产

Refdes Guard

工具层让 get_component("U999") 返回 structured miss；输出层把未验证 refdes-shaped token 包成 ⟨?U999⟩。

Evidence Ledger

规则很硬：No token, no claim. 没有 sch:<file>#<refdes> 这类证据 token 的 finding 在报告前被丢弃。

Tool-use Agent

模型不能直接声明工程事实，必须通过 list_components、get_component、get_nc_pins、search_datasheet 查询。

3 条 active rules

R001/R002/R003 已能展示 EDA 字段检查、硬件字段规范、pin 级结构解析和 reviewer attention allocation。

04

诚实边界

没有专家 gold-label accuracy

不能说 “Hardwise 准确率 xx%”。应该说：Hardwise 在 public corpus 上完成 regression smoke，并跟踪 parser failure、unverified refdes、evidence dropped、finding distribution。

R003数据不漂亮但结构正确。没有 datasheet 证据时输出 reviewer_to_confirm，比编一个 likely_ok 更可信。
R004/R005依赖 schematic-side net parser。当前 PCB-side net parser 是 post-Layout diagnostic，不能作为 pre-Layout 证据。
Cadence当前 MVP 只做 KiCad。Cadence 是授权环境里的新 adapter，不是现在的 submission gate。
05

停工线

停止扩规则数量

暂停 R004/R005/R006/R007。规则数量不是短板，主线是 anti-hallucination + evidence discipline。

停止扩大 eval

没有人工 gold label，再大的 corpus 也只是 smoke，不要包装 accuracy benchmark。

停止产品化入口

GitHub Action、PR bot、Web UI、WebSocket、boardview canvas 都是产品入口，不是 MVP 证明点。

停止跨节点扩张

PCB review、仿真、测试回灌、BOM/PLM、FMEA 都会稀释 pre-layout schematic review 锚点。

06

下一步最小收尾

  1. 重写面试叙事：受约束的 schematic-review Agent，而不是“大硬件 AI 平台”。
  2. 清理 README / PLAN 的扩张语气：R001-R003 是 active scope，R004/R005 是 post-MVP。
  3. 准备 demo script：review、ask U999、eval summary 三段即可。
  4. 只做稳定性收尾：文档、报告显示、测试失败、demo artifact、投递材料。
硬件 AI Agent 的价值不在模型说得多像专家，而在它什么时候被允许说、说的对象从哪里来、证据能不能追溯。
