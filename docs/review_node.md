# 原理图评审节点画像

> Hardwise 整个项目的 demo 锚点。所有取舍都回到"这个节点能不能省工"。
>
> 内容只覆盖**画 PCB 之前的原理图评审**，不延伸到 PCB / 仿真 / 板卡生产 / 整机测试。任何跨节点的诱惑请立即砍掉。
>
> 本文档不涉及任何公司真实工程、料号、客户、项目代号；所有举例使用通用器件类别（"某千兆 PHY"、"某 I2C 扩展器"），不写具体型号。
>
> 节点画像基于 5 年硬件研发经验自述（2026-05-09），后续遇到与文档不符的真实工程实践时回填修订。

---

## 节点位置

硬件研发的典型阶段序列（公开通用模型，非任何特定公司流程）：

```
原理图设计前 → 原理图进 Layout 前 → gerber 前 → 板卡第一次生产前 → 板卡第二次生产前
   (需求清单+方案)   (★原理图评审在此格)   (PCB 评审 + PCB 定稿)
```

**Hardwise 只插在【原理图进 Layout 前】这一格里。**

- **前置阶段**："原理图设计前" — 交付《需求清单+评审记录》和《方案设计+评审记录》
- **本阶段产出（共 8 项）**：软硬件接口文档、Connector_pin_define、FMEA 方案、仿真建议列表、原理图 checklist、原理图检视+评审记录、供应商 Review 原理图记录、原理图+DRC 结果
- **后置阶段**："gerber 前" — PCB 评审 / 仿真 close / 供应商 PCB review / Cable_list / PCB 定稿
- **一次评审，不分两次**：本阶段只组织一次正式原理图评审会

为什么这一格价值最大：硬件不像软件，问题如果到打板、调试、整机阶段才发现，修改代价以"改板 + 验证延期 + 物料浪费"为单位计。越早拦住越便宜，原理图评审是这条曲线最早的可干预点之一。

## 这一格的真实工作流

### 输入物（评审者真正会查的资料）

**只有 2 类**——评审节点是高度闭环的内部动作：

1. **当前原理图（sch）** — 评审主对象
2. **通用原理图 checklist** — 公司维护的统一模板（22 条左右），每项要求"已检查"状态

> "上一代 sch 复用 + delta 重审"是真实评审者的工作习惯，但 MVP 不打算实现"两版 sch 自动 diff"功能，所以**不列为 demo 输入**。

**不在评审输入里的**（这些是评审产出之后才出的下游交付物，曾误列）：
- ❌ 软硬件接口文档 — 原理图完成后才做，给 BMC / BIOS 看（什么挂在哪路 I2C / GPIO 怎么定义）
- ❌ Connector_pin_define — 原理图之后出，主板侧出，用于和挂耳/背板/Riser 卡两边对齐，免得烧板或信号问题
- ❌ FMEA 方案 — 原理图之后出，评估器件失效场景
- ❌ 仿真建议列表 — 原理图之后出，标识哪些高速信号需要仿真

### 资料形态

- 主形态：sch 工具（如 Cadence / KiCad）直接打开 + datasheet PDF
- 沟通形态：基本闭环（与外部团队**不**频繁交互）
- 唯一对外：**供应商 Review 原理图**（邮件原文或截图存档，作为评审记录之一）
- 留痕：SVN 维护原理图最新版本；任何改动（含 ECN）需在 SVN 发布

### 输出物

| 输出 | 字段/形态 |
|---|---|
| **《原理图 checklist》** | 表格；通用模板（22 条），每条状态必须为"已检查" |
| **《SCH_review_feedback_list 汇总表》** | 截图（标问题位置）+ 问题描述（提出方）+ 反馈栏（提出方）+ 是否接受（接收方）+ 不接受时说明原因 / 接受时给反馈和修改动作 + 状态（close）— 评审会上一条条过 |
| **《评审记录》** | 会议纪要 / 评审会签表扫描档 |
| **《供应商 Review 原理图记录》** | 邮件原文或截图存档 |
| **《原理图 + DRC 结果》** | 按 "板卡+VA+日期" 命名上传 SVN；要求所有物料通过 CIS 库调用；第一页有连接器/按键/指示灯位号图；ECN 触发新版本发布 |

> 反馈表字段比想象的轻 — 没有"严重度 / 提出人 / 责任工程师"等强结构；评审者直接在原理图截图上标注，文字描述问题，对方回接受/不接受。

### 评审者最关注的事（按优先级）

1. **新建器件** — 是否新建了 symbol，封装尺寸是否合理，pin 定义是否正确
2. **新建器件对应的 datasheet** — 评审者要重新过一遍这颗芯片的 datasheet
3. **电源裕量** — 电源 pin 数量 / 电流 / 电源通路压降 / 磁珠 DCR / MOS 开启电压等

> 评审者不会从零审视每一颗器件——大量项目实际上从上一代复制 + 重审 delta（这一行为不在 MVP 范围）。

### checklist 来源与维护

- 公司硬件团队维护通用模板（22 条左右）
- Hardwise 已从 22 条中精选 5 条进 `data/checklists/sch_review.yaml`
- 业务线分流情况、维护频率、谁负责 — 待补

## 田野调查 7 问回填（2026-05-09 自述版）

### Q1 节点位置 ✅

原理图评审 = 阶段 "原理图进 Layout 前" 的核心动作；一次评审，不分两次；前置是需求/方案评审，后置是 PCB 评审。

### Q2 输入文件清单 ✅

sch + 通用原理图 checklist。**仅此 2 类**。其它（软硬件接口文档 / Connector_pin_define / FMEA / 仿真建议列表）都是评审产物之后才出的；上一代 sch 是真实习惯但不在 MVP 范围。

### Q3 资料形态 ✅

sch 工具 + datasheet PDF；评审基本闭环；只有供应商 review 那一环对外（邮件原文/截图）；SVN 留痕。

### Q4 输出物模板字段 ✅

《SCH_review_feedback_list》= 截图 + 问题描述 + 反馈栏 + 是否接受 + 不接受原因/接受时反馈修改 + close 状态。比一般想象的轻，没有强结构（无严重度/责任人字段），评审会一条条过。

### Q5 checklist 来源与维护 ⚠️ 部分

公司通用模板（22 条）。维护方/频率/业务线分流待补。

### Q6 最容易漏的问题 ✅

1. 新建器件 symbol/封装
2. 新建器件对应 datasheet 不熟
3. 电源裕量

## Hardwise 与该节点的对位

| 节点动作 | Hardwise 怎么 plug 进去 |
|---|---|
| 评审者要"先列新建器件清单做注意力分配" | **R001 新建器件候选识别**（取 `parse_schematic` 的原始 footprint 字段为空 → 弱信号；**注意不能用 `parse_project` 的 merged BoardRegistry，它会从 .kicad_pcb 回填 footprint，破坏判定**；后续 R001b 加在线库白名单）|
| 评审者翻 datasheet 找规格 | 向量库 + `lookup_datasheet` 工具（Slice 3）|
| 评审者按 22 条 checklist 逐项过 | `check_rule(rule_id)` 工具 + `data/checklists/sch_review.yaml`（5 条精选） |
| 评审者写《SCH_review_feedback_list》 | `report/markdown.py` 输出对齐字段（截图/描述/接受/状态）|
| 评审者引用 datasheet 第 N 页作为依据 | Evidence Ledger（每条 finding 带 `datasheet:xx.pdf#pNN` token） |
| 评审者复用上一代经验 | Sleep Consolidator（候选规则 + 人工 gate） |
| 评审者写 "U23 的去耦电容" | Refdes Guard（`U23` 必须存在于 EDA registry，否则标 `⟨?U23⟩`） |

**Hardwise 不替代评审者的判断**——它把"翻 datasheet → 列新建器件 → 对位号 → 逐条过 checklist → 写报告"这条信息整理链路压缩，让评审者把时间花在判断"这个设计是否正确"上。

## 节点之外（明确 OUT-OF-SCOPE）

- PCB 评审（不同 checklist，不同输入）
- 仿真建议列表（独立流程，归仿真团队）
- FMEA 方案（评审之后产物）
- 软硬件接口文档（评审之后产物）
- Connector_pin_define（评审之后产物）
- 单元测试 / 板卡生产相关（远在评审节点之后）
- **PLM-grade BOM 校验**（BOM 真正带 PLM metadata 要等到 Gerber 之后；评审节点的"BOM"只是 sch 导出的位号清单，不是供应商料号/生命周期/价格那种 BOM）
- BOM 数据治理 / PLM 维护（独立系统）
