Hardwise 原理图检视 Demo

把原理图评审里的查证工作先整理好

输入一个公开 KiCad 工程，可以自动生成检视清单，也可以直接用自然语言追问原理图细节。硬件工程师不用先读代码，就能看到哪些器件、哪些脚位、哪些证据需要重点确认。

扫描器件121来自原理图工程
检视意见84按规则自动整理
NC 脚位77定位到具体位号和管脚
原理图问答可问按位号 / 脚位追问

它解决什么

  • 原理图评审前，要反复对位号、脚位、数据手册和检查表。
  • 很多时间花在搬运和整理，不是判断设计本身。
  • Hardwise 先把可疑点、来源和可追问对象整理出来。

输出长什么样

  • 一份中文网页 / 文本检视报告。
  • 一个可对话的原理图查询入口。
  • 每条意见包含：对象、问题、建议动作、证据位置。

为什么可信

  • 位号来自原理图解析，不让模型自由编。
  • 回答会先查工具结果，再组织成人能读懂的话。
  • 没有证据来源的结论不会进入报告。

也可以直接问原理图

工程师不一定只想看整份报告，也会临时追问：某颗器件是什么、某个脚是否悬空、某个不存在的位号是不是写错了。Hardwise 会先查原理图解析结果，再回答。

工程师问

U4 这颗器件有几个 NC 脚？

Hardwise 答

U4 有 2 个 NC 脚：3 脚 FB-、4 脚 S/S。建议结合数据手册确认这些脚应悬空、接地还是接电源。

工程师问

U3 是什么器件？它有哪些已解析属性？

Hardwise 答

U3 是 7805 稳压器。当前可查到位号、器件值、封装、原理图来源文件等结构化属性；若已导入数据手册，还可以继续查参数页。

工程师问

U999 是什么器件？

Hardwise 答

原理图注册表里没有 U999。系统不会编造器件，会提示确认位号或从相近结果里选择。

报告里的单条意见示例

硬件工程师看到的是评审语言：哪个位号、哪里不完整、下一步确认什么。

需确认 电容耐压标注 位号 C2

C2 的 value 写的是 220uF，但没有标出额定耐压，因此暂时无法判断是否满足 80% 降额要求。

建议补充耐压值，例如 220uF/25V；评审人再结合该网络实际工作电压确认是否合规。

证据位置：sch:pic_programmer.kicad_sch#C2
1. 接入 EDA 数据

解析公开 KiCad 工程，拿到真实器件和脚位清单。

2. 支持对话查询

按位号、器件属性、NC 脚位追问原理图细节。

3. 生成检视清单

覆盖新器件、电容耐压、NC 脚位这类高频检查。

4. 留下可追溯结果

输出报告、结构化库和可继续沉淀的候选规则。

一句话亮点

这个 demo 展示的不是“让 AI 写一段话”，而是把硬件评审流程拆成可查证的步骤：原理图里有什么、工程师问了什么、报告写了什么、每条结论从哪里来，都能追回去。

可复现命令

uv run hardwise review data/projects/pic_programmer --rules R001,R002,R003 --format html uv run hardwise ask data/projects/pic_programmer "U4 这颗器件有几个 NC 脚？"
